# Internationalisation (i18n)

## Why this exists

NM-ZWDS targets two distinct Chinese-speaking markets:

- **Malaysia** — Simplified Chinese (`zh-CN`)
- **Taiwan** — Traditional Chinese (`zh-TW`)

Plus an English-speaking audience. The free-test journey (`/free-test`, `/free-result/:id`, `/free-test-ended`) was the first surface to receive multi-language support. The same system will be extended to the full authenticated app over subsequent sprints.

---

## Architecture overview

```
LanguageContext (React Context)
├── Language type: "en" | "zh-CN" | "zh-TW"
├── isChineseLanguage(lang) → boolean
├── detectInitialLanguage() → Language   (localStorage → browser → "en")
├── changeLanguage(lang)                 (state + localStorage)
└── t(key)                               (dot-path lookup with fallback chain)

Translation files
├── src/translations/en/        (complete)
├── src/translations/zh-CN/     (complete — Simplified, Malaysia)
└── src/translations/zh-TW/     (complete for free-test, stubs elsewhere)

LanguageToggle component
└── Shown in Navbar on /free-test routes only (for now)
```

---

## Language detection logic

On first visit (no localStorage entry), the language is detected from `navigator.language`:

| Browser locale | Detected language |
|---|---|
| `zh-TW`, `zh-HK`, `zh-MO` | `zh-TW` |
| Any other `zh-*` | `zh-CN` |
| Anything else | `en` |

On return visits, the persisted `localStorage["language"]` value is used. Invalid or unrecognised values are ignored and fall through to browser detection.

---

## Fallback chain

When a translation key is missing in the selected language, `t()` walks a fallback chain rather than showing the raw key string. This makes partial translations safe to ship.

| Selected language | Fallback order |
|---|---|
| `en` | en |
| `zh-CN` | zh-CN → en |
| `zh-TW` | zh-TW → zh-CN → en |

The `zh-TW` analysis folder contains only stub translations for the top-level labels. Detailed star interpretation copy falls back to `zh-CN` automatically. Native translators can fill in the full Traditional content without any code changes — just add keys to `src/translations/zh-TW/analysis/`.

## Translation file structure

```
src/translations/
├── en.ts               ← shim re-exporting en/index
├── zh-CN.ts            ← shim re-exporting zh-CN/index
├── zh-TW.ts            ← shim re-exporting zh-TW/index
├── en/
│   ├── index.ts        ← aggregates all namespaces
│   ├── common.ts       ← common.*, general.*
│   ├── freeTest.ts     ← freeTest.*, freeTestEnded.*, freeTestPromo.*
│   ├── calculate.ts    ← calculate.*, form.*
│   ├── chart.ts        ← myChart.*, result.*, pdfExport.*
│   ├── profile.ts      ← profile.*, settings.*
│   ├── auth.ts         ← auth.*
│   ├── navigation.ts   ← navbar.*, nav.*, app.*
│   ├── validation.ts   ← validation.*, errors.*
│   ├── dashboard.ts    ← dashboard.*
│   ├── zwds.ts         ← zwds.* (palace/star names)
│   ├── membership.ts   ← membershipPaused.*
│   ├── misc.ts         ← notFound.*
│   └── analysis/
│       ├── index.ts
│       ├── summary.ts
│       ├── watchout.ts
│       ├── health.ts
│       └── ...
├── zh-CN/              ← mirrors en/ structure, Simplified Chinese
└── zh-TW/              ← mirrors en/ structure, Traditional Chinese
```

The shim files (`en.ts`, `zh-CN.ts`, `zh-TW.ts`) exist so `LanguageContext` can import each language with a single line. Each `index.ts` spreads all namespace modules into one flat object.

## The `isChineseLanguage` helper

**Always use this helper instead of `language !== "en"` or direct comparisons.**

```typescript
import { isChineseLanguage } from "../context/LanguageContext";

const zh = isChineseLanguage(language); // true for "zh-CN" and "zh-TW"
```

Why: if a fourth language is added (e.g. Bahasa Melayu `"ms"`), `language !== "en"` incorrectly returns `true` for Malay. The helper is the single point of update.

For **date locale strings**, use the language code directly since `"zh-CN"` and `"zh-TW"` are valid BCP-47 locale tags:

```typescript
const locale = language === "en" ? "en-US" : language;
date.toLocaleDateString(locale, { year: "numeric", month: "long", day: "numeric" });
```

## How to add a new translation key

1. Add the key to `src/translations/en/<namespace>.ts` first (English is the source of truth).
2. Add the key to `src/translations/zh-CN/<namespace>.ts` (Simplified Chinese).
3. Add the key to `src/translations/zh-TW/<namespace>.ts` (Traditional Chinese).
4. Use `t("namespace.key")` in your component. The fallback chain means you can ship with only the English key if the Chinese translations aren't ready yet — users will see English rather than a raw key string.

**Do not add hardcoded English strings to component files.** Even error messages, loading states, and edge-case fallbacks should go through `t()`.

---

## How to add a new language

1. Add the language code to the `Language` type in `src/context/LanguageContext.tsx`:
   ```typescript
   export type Language = "en" | "zh-CN" | "zh-TW" | "ms"; // example
   ```
2. Update `isChineseLanguage` if the new language is a Chinese variant:
   ```typescript
   export const isChineseLanguage = (lang: Language): boolean =>
     lang === "zh-CN" || lang === "zh-TW";
   ```
   Or add a new helper (e.g. `isMalayLanguage`) if it's a different script.
3. Update `detectInitialLanguage()` with the browser detection logic for the new locale.
4. Update `fallbackChain` with the fallback order for the new language.
5. Update `resources` to include the new language entry.
6. Create `src/translations/<code>/` with all namespace files and an `index.ts`.
7. Create `src/translations/<code>.ts` shim.
8. Add the new option to `LANGUAGE_OPTIONS` in `src/components/LanguageToggle.tsx`.

## Language selector visibility

The `LanguageToggle` component is rendered in `Navbar` (`src/components/navbar.tsx`) but **only on free-test routes** for now:

```typescript
const FREE_TEST_ROUTE_PREFIXES = ["/free-test", "/free-result", "/free-test-ended"];
const isFreeTestRoute = FREE_TEST_ROUTE_PREFIXES.some(
  (prefix) => location.pathname.startsWith(prefix)
);
```

When rolling out to the full app, change the condition to always show the toggle (or remove the condition entirely).

## When to migrate to `react-i18next`

The current custom system is the right choice at this scale. Consider migrating when any of these triggers hit:

| Trigger | Why it matters |
|---|---|
| 4+ languages | Maintenance burden grows; tooling (i18next-parser) becomes worthwhile |
| Interpolation needed | e.g. `"Welcome back, {{name}}"` — word order differs per language |
| Bundle size is a concern | All three language files are bundled for every user; i18next supports namespace lazy-loading |
| External translators | i18next tooling (Phrase, Crowdin) requires JSON format, not TypeScript |
| 5+ components with missing-key bugs | Compile-time key safety (typed resources) becomes worth the setup cost |

Migration path is straightforward: the file structure maps directly to i18next namespaces, and the `t("key")` API is near-identical.
